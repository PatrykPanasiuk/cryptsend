# CryptSend

**Zero-knowledge, end-to-end encrypted secret sharing.**

CryptSend lets you share passwords, tokens, and other sensitive data with a single link. Your secret is encrypted in your browser using AES-256-GCM and embedded directly into the URL — no servers, no databases, no logs. When the recipient opens the link, the secret is decrypted client-side and then removed from the URL.

> **No account required. No data stored. No tracking. Open source.**

---

## Features

- **Client-side encryption** — your secret never leaves your browser unencrypted
- **AES-256-GCM** — industry-standard military-grade encryption
- **Zero-knowledge** — the server never sees your plaintext or decryption key
- **Self-destructing** — the secret is removed from the URL after viewing
- **No account needed** — completely anonymous
- **No storage** — the encrypted payload lives in the URL fragment (hash), not on any server
- **Privacy-first** — no analytics, no cookies, no tracking
- **Dark mode** — respects your system preference
- **Accessible** — keyboard navigable, screen-reader friendly
- **Open source** — MIT licensed, contributions welcome

---

## How It Works

1. **Create** — Type or paste your secret into the text area and click "Create Encrypted Link"
2. **Share** — Send the generated link to your recipient via a trusted channel (e.g., encrypted chat, Signal, WhatsApp)
3. **Reveal** — The recipient opens the link; the secret is decrypted in their browser and displayed once
4. **Self-destruct** — The hash is removed from the URL after the secret is revealed

### Security Model

```
Sender's Browser                  Recipient's Browser
      │                                  │
      │   AES-256-GCM Encrypt            │
      │   ┌─────────────────┐            │
      │   │ Secret + Key →   │            │
      │   │ Encrypted Payload│            │
      │   └─────────────────┘            │
      │         │                         │
      │   URL: https://.../#<payload>.<key>
      │         │                         │
      │         └──── Share Link ────────→│
      │                                   │
      │              AES-256-GCM Decrypt  │
      │              ┌─────────────────┐  │
      │              │ Encrypted        │  │
      │              │ Payload + Key →  │  │
      │              │ Secret           │  │
      │              └─────────────────┘  │
```

- The **key** is a cryptographically random 256-bit value generated by `crypto.getRandomValues()`
- The **encrypted payload** consists of a 12-byte IV + ciphertext + 16-byte GCM authentication tag
- The **key and payload are in the URL fragment** (the part after `#`), which is never sent to the server
- The encryption and decryption use the **Web Crypto API** (`SubtleCrypto`), available in all modern browsers

## Getting Started

### One-Click Deploy

[![Deploy with Vercel](https://vercel.com/button)](https://vercel.com/new/clone?repository-url=https%3A%2F%2Fgithub.com%2FPatrykPanasiuk%2Fcryptsend)

### Manual Deployment

```bash
# Clone the repository
git clone https://github.com/PatrykPanasiuk/cryptsend.git
cd cryptsend

# Install dependencies
npm install

# Install Vercel CLI (if not already installed)
npm i -g vercel

# Deploy to Vercel
vercel --prod
```

### Local Development

```bash
npm install
npm run dev
```

### Server-Side Storage (Optional)

By default, CryptSend stores everything in the URL (fully client-side). If you want to add server-side storage with expiry and self-destruct, set up [Upstash Redis](https://upstash.com) (or any Redis provider) and configure these environment variables:

- `KV_URL` or `REDIS_URL` or `UPSTASH_REDIS_REST_URL`
- `KV_REST_API_TOKEN` or `REDIS_TOKEN` or `UPSTASH_REDIS_REST_TOKEN`

The API routes in `api/secret.mjs` will be enabled automatically when Redis is configured.

## API Reference

### `POST /api/secret`

Store an encrypted payload server-side (requires Redis).

**Request body:**

```json
{
  "encrypted": "<base64url-encoded AES-GCM payload>",
  "ttl": 86400
}
```

- `encrypted` (required) — the base64url-encoded encrypted payload
- `ttl` (optional) — time-to-live in seconds (default: 86400, max: 604800)

**Response** `201 Created`:

```json
{
  "id": "a1b2c3d4e5f6g7h8",
  "ttl": 86400
}
```

### `GET /api/secret?id=<id>`

Retrieve and delete an encrypted payload (one-time read).

**Response** `200 OK`:

```json
{
  "encrypted": "<base64url-encoded AES-GCM payload>"
}
```

**Response** `404 Not Found` (already viewed or expired).

## Security

- **Encryption:** AES-256-GCM using the Web Crypto API (`SubtleCrypto`)
- **Key generation:** Cryptographically secure random bytes (`crypto.getRandomValues()`)
- **Key length:** 256 bits
- **IV length:** 96 bits (12 bytes)
- **Fragment-only keys:** Decryption keys are in the URL fragment — never sent to servers
- **No persistence:** No cookies, localStorage, or IndexedDB used
- **CSP:** Content Security Policy restricts scripts and connections

Report security vulnerabilities to [security@patrykpanasiuk.com](mailto:security@patrykpanasiuk.com).
